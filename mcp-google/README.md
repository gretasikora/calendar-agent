# Calendar Agent MCP Server

A vendored, calendar-only MCP server used by the [calendar-agent](../README.md) project. It wraps the **Google Calendar API** over OAuth2 and exposes a small HTTP endpoint that the Flask API calls.

It is based on the open-source [`@cocal/google-calendar-mcp`](https://github.com/199-mcp/mcp-google) server, trimmed down to just the Calendar functionality this project needs.

## What it provides

Two ways to run:

- **HTTP server** (`npm run http-server`) — the production entry point. Exposes:
  - `GET /health` — health check
  - `POST /mcp/calendar` — accepts `{ user_id, action, params }` and proxies to the calendar tools
- **stdio MCP server** (`npm run start`) — standard MCP transport for MCP clients.

Supported calendar actions: `list-calendars`, `list-events`, `search-events`, `list-colors`, `create-event`, `update-event`, `delete-event`, `get-freebusy`.

## OAuth scope

The server requests a single scope: `https://www.googleapis.com/auth/calendar`.

Tokens are loaded, in order of priority:

1. `GOOGLE_CALENDAR_TOKENS` environment variable (used on Railway) — a single-line JSON blob. The `refresh_token` in this variable is the source of truth; refreshed access tokens are held in memory only.
2. A token file at `~/.config/google-calendar-mcp/tokens.json` (created by `npm run auth`).

The access token (1 hour) is validated and refreshed automatically on every request using the long-lived refresh token.

## Credentials

Provide OAuth client credentials via either:

- `GOOGLE_CLIENT_ID` and `GOOGLE_CLIENT_SECRET` environment variables, or
- a `gcp-oauth.keys.json` file in this directory (Desktop app credentials from Google Cloud Console), or
- a path set in `GOOGLE_OAUTH_CREDENTIALS`.

## Commands

```bash
npm install          # install dependencies (Node v20 LTS recommended)
npm run build        # typecheck + bundle src/ into build/
npm run auth         # run the OAuth flow, saving tokens locally
npm run http-server  # start the HTTP server (default port 3000)
npm test             # run the vitest suite
```

## Deployment notes

- The Railway Dockerfile builds `build/` from `src/` on deploy. The committed `build/` output is regenerated by `npm run build`.
- After re-authenticating, use `node format-tokens-for-railway.js` to produce the single-line JSON for the `GOOGLE_CALENDAR_TOKENS` variable.

## Troubleshooting

**`invalid_grant` / authentication errors** — the refresh token has expired or been revoked. Re-run `npm run auth`, then update `GOOGLE_CALENDAR_TOKENS` on Railway and redeploy. To avoid Testing-mode expiry, publish your OAuth app in the Google Cloud Console.

**Redirect URI mismatch** — the OAuth flow uses `http://localhost:3000/oauth2callback`. Ensure this matches your OAuth client configuration.
